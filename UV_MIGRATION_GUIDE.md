# uv Migration Guide — Blueprints App

## Executive Summary

This document outlines the migration from pip-based package management to **uv**, a modern, fast Python package manager and project tool runner. The benefits include:

- **10-100x faster** dependency installation and syncing
- **Reproducible builds** across all fleet nodes via `uv.lock` (committed to git)  
- **Simplified tooling** — uv + ruff handle both package management and code quality
- **Modern Python packaging** — leverages `pyproject.toml` instead of `requirements.txt`
- **No breaking changes** — app code is unchanged; only installation method differs

See [TESTING_AND_DEPLOYMENT.md](./TESTING_AND_DEPLOYMENT.md) in the private docs for detailed test results, rollback procedures, and deployment strategies specific to your infrastructure.

## Current State (Pre-Migration)

### Package Management
- **Tool:** pip
- **Config file:** `requirements.txt` (8 lines, pinned versions)
- **Lock file:** None (exact versions picked at install time)
- **Speed:** Slow (15-30s+ for dependency resolution)

### Code Quality Tools
- **Linting:** `ruff` (system-wide via setup-python-dev-tools.sh)
- **Formatting:** `ruff format` (same)
- **Config:** `ruff.toml` (repository root)

## Proposed State (Post-Migration)

### Package Management
- **Tool:** uv (system-wide installation via setup-python-dev-tools.sh)
- **Config file:** `pyproject.toml` (PEP 517/518 standard)
- **Lock file:** `uv.lock` (TOML, 1889 lines, **committed to git**)
- **Venv location:** `/root/xarta-node/.venv` (project root — uv convention)
- **Speed:** 10-100x faster
- **Reproducibility:** All nodes install identical versions via lock file

### Code Quality Tools
- **Linting:** `ruff` (unchanged, system-wide)
- **Formatting:** `ruff format` (unchanged)
- **Config:** Migrated from `ruff.toml` → `pyproject.toml[tool.ruff]` (consolidated)

## Files Created (Not Yet Committed)

1. **`pyproject.toml`** — New project configuration (replaces requirements.txt usage)
   - Defines dependencies, optional dev dependencies, and tool configs
   - No build backend (application, not distributable package)
   - Includes full ruff config under `[tool.ruff]`

2. **`uv.lock`** — Dependency lock file (must be committed)
   - Generated by `uv lock` from pyproject.toml
   - Contains exact versions and hashes for all 87 resolved packages
   - Ensures reproducible installs across fleet

3. **`setup-blueprints-uv.sh`** — Alternative setup script for uv-based install
   - Runs `uv sync --frozen` instead of `pip install -r requirements.txt`
   - Creates `.venv` in project root (uv convention)
   - Includes backward-compatibility notes

4. **`blueprints-app/blueprints-app-uv.service.template`** — Updated systemd template
   - Points to `/root/xarta-node/.venv/bin/uvicorn` instead of `/opt/blueprints/venv/bin/uvicorn`
   - Drop-in replacement for `blueprints-app.service.template`

## Migration Paths

### Option A: Full Migration (Recommended)
1. Commit: `pyproject.toml`, `uv.lock`, `setup-blueprints-uv.sh`, and `blueprints-app/blueprints-app-uv.service.template`
2. Update fleet: Run `setup-blueprints-uv.sh` on all nodes
3. Update CI/deployment: Point to new setup script
4. Retire: Keep old `requirements.txt` and `setup-blueprints.sh` for reference for 2-3 releases

### Option B: Phased Migration per-node
1. Test on one or two non-critical nodes first
2. Monitor for issues
3. Fleet-wide once confident

### Option C: Dual Support (Longest Transition)
1. Commit new files (_all_ the above)
2. Keep both `setup-blueprints.sh` (pip) and `setup-blueprints-uv.sh` (uv)
3. Users choose which to run
4. Phase out old script after stable period

## Important Notes

### Lock File Discipline

The `uv.lock` file **must be committed to git** and pushed to the fleet like any code change:

```bash
# After any dependency update:
uv lock                    # Regenerate lock from pyproject.toml
git add pyproject.toml uv.lock
git commit -m "Update dependencies: fastapi..."
bash bp-nodes-push.sh      # Ensure .nodes.json is pushed first
fleet-pull-public.sh       # Pull to all fleet nodes
```

On each node after pull:
```bash
cd /root/xarta-node
uv lock --check            # Verify lock integrity; fails if stale
uv sync --frozen           # Install exactly what's in uv.lock
```

### Venv Location Change

If you choose Option A or C, the venv moves from `/opt/blueprints/venv` to `/root/xarta-node/.venv`:

- **Pro:** Cleaner (follows Python convention of `.venv` in project root)
- **Pro:** Self-contained (easier to migrate or backup)
- **Pro:** Easier to re-generate from lock file
- **Con:** Requires systemd service template update
- **Con:** More disk space in the git working directory (~500MB)

**Note:** You can revert to `/opt/blueprints/venv` if desired by uncommenting the backward-compat symlink in `setup-blueprints-uv.sh`.

### Backward Compatibility

- Old `setup-blueprints.sh` remains unchanged and functional
- Can run either script; uv script takes precedence if `pyproject.toml` is present
- `ruff.toml` remains in repo (not removed) — `pyproject.toml[tool.ruff]` is preferred but not exclusive

## Risks and Mitigation

| Risk | Likelihood | Mitigation |
|------|------------|-----------|
| Lock file conflicts on pull | Low | `.venv` and `uv.lock` are both git-tracked; standard merge conflict handling |
| Dependency version divergence if lock ignored | Medium | Add CI check: `uv lock --check` on every PR |
| Missing `uv.lock` breaks new node setup | High | **Always run `uv lock` before commit** |
| Stale pyproject.toml → wrong versions | Medium | Keep pyproject.toml simple; prefer version pinning in lock file |
| Cross-platform issues (lock file) | Very Low | uv lock files are platform-agnostic; tested on Linux/x86_64 |

## Rollback Plan

If issues occur after migration, refer to private infrastructure documentation for node-specific restore procedures.

For immediate rollback on any node:
```bash
# Disable uv setup
systemctl stop blueprints-app
rm -rf /root/xarta-node/.venv
# Restore to pip-based setup (keep both scripts for transition period)
bash setup-blueprints.sh
systemctl restart blueprints-app
```
